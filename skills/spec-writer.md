# Spec Writer Skill — 人机协作 Spec 编写助手

## Skill 元信息

| 属性 | 值 |
|:---|:---|
| **名称** | `spec-writer` |
| **版本** | 2.0.0 |
| **类型** | 人机协作 Skill |
| **配套文档** | 《企业级 Spec Coding 标准化手册 V5 人机协同版》（13 模块 + 4 Gates） |

---

## 触发条件

当用户提出以下请求时使用此 Skill：

- "帮我写一个 Spec"
- "生成一份 Spec 文档"
- "我要开发一个新功能，帮我做 Spec"
- "基于这个需求写 Spec"

---

## 核心理念

> **Spec 是 AI Coding 的第一生命线。AI 负责组织结构和辅助描述，人负责提供真实信息和最终审核。**
>
> 此 Skill 绝不臆造接口名、类名、方法签名、配置值。凡是需要真实代码库信息的部分，必须停下来向人提问。

### V5 新增能力

此版本覆盖 V5 标准化手册的全部 **13 个模块**，新增：
- **3.4 隐形边界声明**：六维度（幂等/并发/分布式事务/错误分类/时间语义/数据安全）
- **3.10.1 Runtime Profile**：框架/中间件版本矩阵
- **Per-Step Injection**：实施计划中每步的上下文注入列
- **3.12 审查清单**：AI 代码审查清单
- **3.13 部署契约**：容器/K8s/CI 部署配置
- **Gate 4 测试质量门禁**：四道 Quality Gates

## Step 1: 理解需求 + 初始化 Spec 结构

### AI 操作

1. 阅读用户提供的需求描述
2. 生成 Spec ID（格式：`SPEC-{项目缩写}-{序号}`）
3. 创建 Spec 框架结构（按照 V5 模板的 **13 个模块**：3.0 ~ 3.13）
4. 填写 `3.0 元数据` 中可推断的字段

### 🛑 重要：需求不足时先反问

如果用户的需求描述不足 3 句话，或缺少以下任一关键信息：
- 具体的业务场景（在什么情况下触发？）
- 涉及的平台或系统（天猫？京东？中台？）
- 预期的业务量级（订单量？频率？）

**不要进入后续步骤。先反问用户澄清**，直到信息足够支撑 Spec 编写。

示例反问：
```
在开始编写 Spec 之前，我需要先了解几个关键信息：
1. 这个功能的具体业务场景是什么？（例如：什么情况下需要自动发货？）
2. 涉及哪些系统或平台？（例如：天猫 API、京东 API、OMS 中台？）
3. 预期的业务量级是怎样的？（例如：每天多少订单？）
请补充以上信息后，我再继续。
```

### AI 输出示例

```markdown
我已为你创建 Spec 框架：`SPEC-ANTA-001`。

V5 模板包含 13 个模块（3.0 元数据 ~ 3.13 部署契约），4 道 Quality Gates。

在开始填写之前，请确认以下信息：
1. 这个需求的关联 Jira/TAPD 编号是什么？
2. 涉及哪些微服务模块？（如 oms-order-service, platform-adapter 等）

✅ 以下是我已填写的内容：
- Spec ID: SPEC-ANTA-001
- 状态: Draft

⬜ 以下需要你补充：
- 关联需求编号
- 关联系统列表
```

## Step 3: 填写 3.2 存量事实基准

### 🛑 红线：AI 不自行填写

**此模块是防止 AI 破坏现有系统的关键防线。AI 绝不自行推断哪些模块不能动。**

### AI 操作

向用户提出以下问题：

```markdown
🛑 存量事实基准（Fact Base）必须由你（TL 或资深开发）填写。

我不能替你填写，因为：
- 我看不到完整的代码库
- 我不知道哪些代码有隐藏的历史依赖
- 我可能遗漏未文档化的副作用

请回答以下问题：

1. 禁止变更清单：
   - 哪些类/方法/表绝对不能修改？
   - 为什么不能修改？（对账？审计？历史数据兼容？）
   - 误改会影响什么？

2. 已有接口清单：
   - 当前系统中已经有哪些与需求相关的查询/操作接口？
   - 它们的调用方式是什么？

3. 消息队列依赖：
   - 当前系统使用什么消息中间件？
   - 哪些已有的 Topic/Event 与本次需求相关？

请以表格形式提供，格式参考 V5 模板 3.2 节。
```

### AI 输出

将用户提供的信息填入 3.2 模板，不添加任何用户未提及的内容。

```markdown
☐ 修改的模块列表是否完整？
☐ 极端场景处理策略是否明确？（失败重试？转人工？回滚？）
☐ 是否与 Fact Base 的红线有冲突？
```

## Step 5: 填写 3.4 隐形边界声明 🆕

### 🛑 红线：AI 不自行填写

**隐形边界是最隐蔽的幻觉类型——代码能编译、测试能过，但上线就炸。AI 绝不可能自己"猜"对这些约束。**

### AI 操作

逐维度向用户提问，六维度全部覆盖：

```markdown
🛑 隐形边界声明（3.4）是 V5 新增模块，也是上线事故的高发区。

我不能替你填写，因为：
- 我不知道 Redis 缓存键的命名约定
- 我不知道哪些操作被视为可重试 vs 不可重试
- 我不知道敏感数据的加密算法和安全策略
- 我不知道分布式事务的补偿机制设计

请在以下六个维度中逐一确认：

1️⃣ 幂等性：
   - 哪些写操作需要幂等？幂等键是什么？
   - 重复调用时的预期行为是什么？（返回已有记录？报错？幂等成功？）

2️⃣ 并发安全：
   - 哪些共享资源有并发风险（库存？流水记录？）？
   - 锁策略是什么？（行级锁？乐观锁？分布式锁？）
   - 唯一索引在哪些字段上？

3️⃣ 分布式事务：
   - 哪些操作涉及跨系统（DB + API + MQ）？
   - 最终一致性方案是什么？（本地事务表 + 补偿？SAGA？TCC？）
   - 补偿机制是什么？

4️⃣ 错误分类：
   - 哪些错误是可重试的？（网络超时？限流？）
   - 哪些错误不可重试？（参数错误？业务规则拒绝？）
   - 幂等成功应如何处理？

5️⃣ 时间语义：
   - 时间字段用什么类型？（Instant？LocalDateTime？Date？）
   - 时区策略是什么？（统一 UTC？按用户时区？）
   - 超时/过期判断使用相对时间还是绝对时间？

6️⃣ 数据安全：
   - 哪些字段是敏感数据？脱敏策略是什么？
   - 日志中允许输出什么？严禁输出什么？
   - 哪些字段需要加密存储？加密算法是什么？
```

### 👤 确认点

```markdown
⚠️ 幻觉提醒：
隐形边界的六个维度必须全部填写。任何一个留空，AI 实现时就会用"网上最常见的做法"来填——这大概率不是你们项目的做法。

请逐维度确认：
☐ 每个涉及数据库写操作的步骤，幂等性是否已声明？
☐ 每个涉及共享资源修改的步骤，并发锁策略是否已声明？
☐ 每个跨系统操作，分布式事务方案是否已声明？
☐ 每个调用外部接口的步骤，错误分类是否已声明？
☐ 所有时间相关字段，时区策略是否已声明？
☐ 所有可能出现在日志中的数据，脱敏策略是否已声明？
```

### AI 输出

将用户提供的信息严格按照六维度格式填入 3.4 模板。

## Step 6: 填写 3.5 CodeAnchor（含 V2 增强）

### 🛑 这是幻觉最高危区域

**AI 绝不自行填写任何接口的类名、方法签名、DI 注解、真实调用示例。因为这些信息只能从真实代码库中获取。**

### AI 操作

1. 列出需要锚定的逻辑接口（基于需求推理的"逻辑名称"）
2. 指导用户如何搜索代码库
3. **V2 新增：引导用户提供完整方法签名（含 throws）、返回类型、DI 注解、调用示例**

```markdown
🛑 CodeAnchor（3.5）是 Spec 中最重要的字段，也是 AI 最容易产生幻觉的地方。

我不能替你填写 CodeAnchor 中的：
- 真实类全路径（因为我无法访问你的代码库）
- 真实方法签名（同上）
- 返回值类型（同上）
- DI 注解方式（同上）
- 已有调用示例（同上）

我能做的是：
✅ 列出需要在代码库中搜索的"逻辑接口"
✅ 告诉你用什么关键词搜索
✅ 帮你整理搜索结果到 CodeAnchor 表格

V2 增强要求（防止代码映射幻觉）：
每个 CodeAnchor 除了基本字段外，还需提供：
✅ 完整方法签名（含 throws 声明）
✅ 返回值类型完整定义
✅ 所在类的 DI 注解（@Service / @Component / @Repository）
✅ 项目中已有的至少 1 处真实调用示例

---

根据你的需求，你需要在代码库中搜索以下逻辑接口：

1. 订单查询（需要根据订单号查询订单详情）
   → 搜索关键词: "OrderQuery", "queryById", "getOrder"
   → 预期在模块: oms-order-service

2. 订单拆分（需要将订单拆分为子订单）
   → 搜索关键词: "split", "divide", "order" + "split"
   → 预期在模块: oms-order-service

3. 平台发货（需要调用天猫/京东发货 API）
   → 搜索关键词: "virtualShip", "noLogistics", "ship"
   → 预期在模块: platform-adapter / platform-integration

---

操作步骤：
1. 在 Kiro/Cursor 中搜索上述关键词
2. 找到真实的类和方法
3. 确认：方法签名（含 throws）→ DI 注解 → 返回值类型 → 调用示例
4. 将结果告诉我，我帮你填入 CodeAnchor V2 表格

⚠️ 如果搜索不到某个逻辑接口，说明这个接口需要新建。
   那你需要明确告诉我：新接口的命名、所在模块、期望的方法签名。
```

### AI 输出

将用户提供的真实接口信息填入 CodeAnchor V2 模板（含注入清单列），不做任何修改。

## Step 9: 填写 3.9-3.10 实施计划 + 环境配置（含 Runtime Profile + Per-Step Injection）

### 3.9 实施计划 + Per-Step Injection 🆕

```markdown
⚠️ 幻觉提醒：
以下实施计划由我根据需求推理生成。
Step 的拆分粒度、验证命令是否正确，需要你确认。

V5 新增：每个 Step 还需填写"本 Step 必须注入的上下文"列。
这是防止上下文衰减幻觉的关键——长对话中早期注入的信息可能已被 AI 遗忘。

请逐项确认：
☐ 每个 Step 是否可独立编译/测试验证？
☐ Step 之间的依赖关系是否正确？
☐ "本 Step 必须注入的上下文"是否完整？
    （如：涉及 ANC-01 的 Step，是否将 ANC-01 的源文件片段列入注入清单？）
☐ 注入顺序是否正确？（Runtime Profile → CodeAnchor 源文件 → 项目规范 → 本 Step 特殊要求）
☐ 验证命令（如 mvn test -Dtest=xxx）在当前项目中是否可执行？
☐ 是否遗漏了必要的环境准备步骤？（如数据库迁移、配置中心变更）
```

### 3.10 环境配置

### 🛑 环境配置值的红线

```markdown
🛑 环境配置值是 AI 幻觉的另一个高发区。
我绝不填写以下内容：
- 数据库连接 URL
- API Key / Secret
- 服务器 IP / 端口
- RocketMQ Topic / Group 名称
- 配置中心的具体引用路径

请由 DevOps 提供真实的环境配置信息。

我能做的：
✅ 列出需要配置的项
✅ 建议使用配置中心引用的方式（如 Apollo、K8s ConfigMap）
✅ 提醒密钥类配置必须通过密钥管理服务获取

---

请确认以下配置项的真实值或配置中心引用：
☐ 数据库连接引用: ${????}
☐ 天猫 API Key 引用: ${????}
☐ RocketMQ Topic 名称: ????
☐ 重试次数和间隔: ????
```

### 3.10.1 Runtime Profile 🆕


### AI 绝不可以做

| 🛑 绝不可以 | 原因 |
|:---|:---|
| 自行填写 CodeAnchor 中的类名和方法签名 | 无法访问用户代码库，必定臆造 |
| 自行填写第三方 API 字段定义 | 无法访问官方 API 文档，必定臆造 |

| 自行判定 Fact Base 红线 | 不了解代码库历史，必定遗漏 |

| 自行填写 Runtime Profile 版本 | 无法读取 pom.xml，必定臆造 |
| 自行填写部署契约真实值 | 无法获取生产环境配置 |
| 自行签字通过任何 Quality Gate | 签字是人的责任，AI 不能替代 |
| 声称 Spec 已准备好进入实现 | 必须由人确认四道 Gate 通过 |
