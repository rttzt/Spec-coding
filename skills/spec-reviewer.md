# Spec Reviewer Skill — 人机协作 Spec 审查助手

## Skill 元信息

| 属性 | 值 |
|:---|:---|
| **名称** | `spec-reviewer` |
| **版本** | 2.0.0 |
| **类型** | 人机协作 Skill |
| **配套文档** | 《企业级 Spec Coding 标准化手册 V5 人机协同版》（13 模块 + 4 Gates） |

---

## 触发条件

当用户提出以下请求时使用此 Skill：

- "帮我 review 这个 Spec"
- "审查这份 Spec 文档"
- "检查 Spec 有没有问题"
- "Spec 写完了，帮我看看能不能通过 Quality Gates"

---

## 可靠性声明

> **此 Skill 的设计严格区分"AI 可验证"和"必须人工验证"。**
>
> - ✅ 结构性和一致性检查：由 AI 执行，结果可靠（模式匹配，无幻觉风险）。
> - ⚠️ 矛盾标记：由 AI 检测并标记为"潜在问题"，必须由人判断。
> - 👤 真实性验证：AI 绝不执行，仅生成人类审核清单。
>
> **此 Skill 绝不在需要真实世界知识的问题上给出"通过"或"不通过"的结论。**

### V5 新增审查能力

此版本覆盖 V5 标准化手册的全部 **13 个模块** 和 **4 道 Quality Gates**：
- 审查 **3.4 隐形边界**（六维度完整性）
- 审查 **3.5 CodeAnchor V2**（含注入清单列）
- 审查 **3.10.1 Runtime Profile**（版本矩阵一致性）
- 审查 **3.9 Per-Step Injection**（每步上下文注入完整性）
- 审查 **3.12 审查清单**（AI 检查项 vs 人工检查项分离）
- 审查 **3.13 部署契约**（AI 禁止修改标记完整性）
- 审查 **Gate 4 测试质量门禁** 就绪状态
- 与 `spec-tester` Skill 的协作路径

---

## 审查流程

```
输入 Spec 文档
    │
    ▼
┌───────────────────────────────────────────────────────────┐
│ Pass 1: 完整性检查（全部由 AI 执行）                        │
│ → 检查 13 个模块是否存在且非空                              │
│ → 输出: ✅ 通过 / ❌ 缺失项列表                             │
├───────────────────────────────────────────────────────────┤
│ Pass 2: 格式校验（全部由 AI 执行）                          │
│ → 输出: ✅ 格式正确 / ❌ 格式问题列表                        │
├───────────────────────────────────────────────────────────┤
│ Pass 3: 交叉引用一致性（全部由 AI 执行）                     │
│ → 输出: ✅ 一致 / ⚠️ 不一致项列表                           │
├───────────────────────────────────────────────────────────┤
│ Pass 4: 矛盾检测（AI 标记 + 人判断）                        │
│ → 新增: 隐形边界与增量变更的矛盾检测                         │
│ → 输出: ⚠️ 潜在矛盾列表 → 请人确认                          │
├───────────────────────────────────────────────────────────┤
│ Pass 5: Quality Gates 就绪检查（AI 标记 + 人判断）          │
│ → 新增: Gate 4 测试质量门禁就绪检查                         │
│ → 输出: 各 Gate 审核项完成度 → 请对应角色确认               │
├───────────────────────────────────────────────────────────┤
│ Pass 6: 人工审核清单生成（全部为提醒，AI 不执行）            │
│ → 新增: DevOps 对 3.13 部署契约的审核任务                   │
│ → 新增: QA 用 spec-tester 审计测试质量的任务                │
│ → 输出: 👤 每个角色的审核任务清单                            │
└───────────────────────────────────────────────────────────┘
    │
    ▼
输出审查报告
```

## Pass 1: 完整性检查

### 检查逻辑

AI 逐一检查 V5 模板的 **13 个模块** 是否存在且非空。

### 检查项

```
[  ] 3.0 元数据: Spec ID、版本、状态、作者
[  ] 3.1 背景与目标: 业务背景、核心价值、量化指标
[  ] 3.2 存量事实基准: 禁止变更清单、存量接口清单、消息队列依赖
[  ] 3.3 增量变更描述: 新增、修改、废弃、极端场景处理
[  ] 3.4 隐形边界声明 🆕:
     [  ] 3.4.1 幂等性: 操作/幂等键/重复调用行为 表非空
     [  ] 3.4.2 并发安全: 共享资源/并发风险/锁策略/锁粒度 表非空
     [  ] 3.4.3 分布式事务: 跨系统操作/事务边界/最终一致性方案/补偿机制 表非空
     [  ] 3.4.4 错误分类: 错误类型/示例错误码/处理策略/最大重试次数 表非空
     [  ] 3.4.5 时间语义: 时间字段/存储类型/时区策略/比较方式 表非空
     [  ] 3.4.6 数据安全: 敏感字段/脱敏策略/日志输出规则/存储加密 表非空
[  ] 3.5 CodeAnchor V2: 含逻辑名称、真实类路径、方法签名、返回值类型、DI 注解、注入清单 🆕
[  ] 3.6 第三方 API: 至少包含 API 名称和 Request/Response 结构
[  ] 3.7 数据模型: 新增表 DDL 或修改表字段定义
[  ] 3.8 TDD 规划: Given-When-Then 场景、TDD 执行步骤
[  ] 3.9 实施计划: 逐 Step 的任务描述、验证方式、Per-Step Injection 列 🆕
[  ] 3.10 环境配置: 配置项列表、Runtime Profile 版本矩阵 🆕
[  ] 3.11 质量门禁: 四道 Gate 的签字表格 🆕
[  ] 3.12 审查清单 🆕: AI 可检查项 + 人工需检查项
[  ] 3.13 部署契约 🆕: 容器/K8s/CI 配置 + AI 禁止修改标记
```

## Pass 3: 交叉引用一致性

### 检查逻辑

这是**最核心、最可靠的 AI 检查**——检查 Spec 内部的交叉引用是否自洽。

> ⚠️ **前提条件**：此 Pass 的交叉引用检查依赖 Spec 使用**规范化的 ID 引用格式**（如 `ANC-01`、`RED-01`、`API-01`）。如果 Spec 中的引用使用自然语言描述（如"Mock 订单拆分的那个接口"）而非规范化编号，交叉引用的字符串匹配将无法工作，检查可能遗漏。
>
> 如果 Spec 中未使用规范化编号，此 Pass 将输出以下警告：**"检测到 Spec 未使用规范化 ID 引用格式，交叉引用一致性检查无法执行。建议在 spec-writer 阶段强制生成 ANC/RED 编号。"**

## Pass 4: 矛盾检测

### ⚠️ 重要说明

> 此 Pass 由 AI 检测文本层面的潜在矛盾。**AI 标记的"矛盾"可能是真矛盾，也可能是 AI 理解偏差。所有标记项必须由人最终判断。**

### 检查项

```markdown
## ⚠️ Pass 4: 矛盾检测（需人工确认）

| # | 矛盾描述 | 涉及模块 | 建议人工确认 |

| 2 | 3.2 未列出财务模块相关红线，但需求背景提到"不影响财务结算"。是否遗漏了财务模块的红线？ | 3.2 | ⚠️ 请 TL 确认：是否需要将财务相关类也加入禁止变更清单？ |
```

## Pass 6: 人工审核清单

### 👤 此 Pass 全部是对人类的提醒，AI 不执行任何验证。

```markdown

> 以下检查项 AI 无法验证。必须由对应角色在真实环境中逐项确认。

### PM 必须检查

☐ 确认 3.1 业务背景和量化指标是否与原始需求一致？
```

## 关键行为准则

### AI 可以做（零幻觉风险）

| ✅ 可以做 | 原理 |
|:---|:---|
| 检查模块是否完整（含子模块） | 计数和空值检测 |
| 检查交叉引用是否自洽 | 字符串匹配 |
| 检查 Markdown/YAML 格式 | 语法解析 |
| 标记文本层面的潜在矛盾 | 语义对比，标注为"潜在" |
| 检查 Per-Step Injection 列是否已填写 | 空值检测 |
| 检查部署契约中 AI 禁止修改标记是否完整 | 模式匹配 |
| 生成人工审核清单 | 基于模板推导 |

### AI 绝不可以做

| 🛑 绝不可以 | 原因 |
|:---|:---|
| 声称 CodeAnchor 已通过验证 | 无法访问用户代码库 |
| 声称 API 字段与文档一致 | 无法访问官方文档 |
| 声称 Fact Base 红线完整 | 不了解代码库历史 |
| 声称 隐形边界 正确 | 不了解项目运行约定 |
| 声称 Runtime Profile 版本正确 | 无法读取 pom.xml |
| 声称业务逻辑正确 | 不理解真实业务上下文 |
| 声称部署契约中配置值正确 | 无法访问生产环境 |
| 签字通过任何 Quality Gate | 签字是人的责任 |
| 说"此 Spec 已就绪" | 必须由人确认四道 Gate 通过 |
| 说"测试通过了" | 必须由 QA 用 spec-tester 审计 + 亲自运行 |
