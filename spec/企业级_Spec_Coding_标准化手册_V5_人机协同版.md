# 企业级 Spec Coding 标准化手册 (V5.0) — 人机协同版

---

## 0. 核心哲学

> **Spec 是 AI Coding 的第一生命线。Spec 的质量决定了 AI 生成代码的质量上限。**
>
> 每一个角色必须在 Spec 环节对他所负责的内容进行把关和兜底。AI 负责执行，人负责审核。没有经过人工审核签字的 Spec，不允许进入 AI 实现阶段。

### V5 与 V4 的关键变化

| 维度 | V4 TDD 融合版 | V5 人机协同版 |
|:---|:---|:---|
| 核心理念 | Spec + TDD | Spec 是第一生命线，人机协同 |
| 接口引用 | 描述性接口名（可能是臆造的） | **CodeAnchor** 锚定真实代码的类和方法 |
| 审核机制 | 建议性角色分工表 | **Quality Gates** 三道强制签字门禁 |
| 上下文管理 | 无 | **Context Injection Checklist** 必注入清单 |
| 验证方式 | 一次性生成 | **Incremental Verification** 逐步骤编译验证 |
| 角色定位 | AI 主导 | **人把关兜底，AI 执行** |
| 永久上下文 | 无 | **AGENTS.md** 项目级 AI 编码规范，自动加载 |

### 上下文分层架构（V5 核心创新）

> **上下文分为两层：永久层（AGENTS.md）和临时层（Spec 注入清单）。**
>
> - **永久层**：技术栈版本、代码规范、永久红线、项目结构——AI 每次对话自动读取，人不需要每次重复准备。
> - **临时层**：本需求的 CodeAnchor、专用 API 字段、新增表 DDL——每个 Spec 特例化，由人按需注入。
>
> 这个分层架构将上下文准备的工作量从"每次都重新来"降低到"维护一次 AGENTS.md + 每个 Spec 补充特例信息"。

```
┌─────────────────────────────────────────┐
│         AGENTS.md（永久层）                │
│  项目技术栈 · 代码规范 · 永久红线 · 目录结构   │
│  ⬆ AI 每次对话自动加载，不需要人每次注入       │
├─────────────────────────────────────────┤
│         Spec 注入清单（临时层）              │
│  本需求的 CodeAnchor · 专用 API · 新增表 DDL  │
│  ⬆ 每个 Spec 特例化，由人按需注入            │
└─────────────────────────────────────────┘
```

## 1. 角色职责与审核矩阵

每一个 Spec 环节都有明确的负责角色。**该角色必须签字确认后，Spec 才能进入下一环节。**

| Spec 模块 | 负责角色 | 核心职责 | 审核要点 |
|:---|:---|:---|:---|
| **背景与目标** | 👤 产品经理 (PM) | 定义业务价值和量化指标 | 业务场景是否真实？量化指标是否可衡量？ |
| **存量事实基准 (Fact Base)** | 👤 技术负责人 (TL) | 划定红线、提供真实约束 | 禁止变更范围是否完整？隐藏副作用是否暴露？ |
| **增量变更描述 (Delta Changes)** | 👤 PM + TL | 描述新增/修改/废弃的逻辑 | 边界条件是否明确？回滚策略是否定义？ |
| **隐形边界声明 (Invisible Boundaries)** | 👤 TL | 声明幂等/并发/事务/错误/时间/安全边界 | 每个维度是否已明确？是否与已有系统约定一致？ |
| **技术契约 — CodeAnchor** | 👤 后端开发 | 锚定真实接口的类和方法签名 | 每个接口是否能在代码库中找到？签名是否匹配？ |
| **技术契约 — 第三方 API** | 👤 后端开发 | 核对官方 API 文档 | 字段名、枚举值、认证方式是否与文档一致？ |
| **技术契约 — 数据模型** | 👤 后端开发 + DBA | 定义新增/修改的字段 | 字段类型、长度、索引是否与数据库对齐？ |
| **验收标准 — Given-When-Then** | 👤 测试工程师 (QA) | 覆盖所有业务分支 | 异常场景是否覆盖？边界条件是否测试？ |
| **TDD 执行规划** | 👤 测试工程师 + 开发 | 定义测试类名和方法名 | 测试是否可独立编译？Mock 对象是否基于真实接口？ |
| **实施计划 — Step 验证点** | 👤 TL | 拆分为原子步骤，定义每步验证方式 | 每个 Step 是否可独立编译/测试验证？ |
| **环境与配置** | 👤 运维/DevOps | 提供真实配置信息 | 服务器路径、密钥引用、中间件版本是否真实？ |
| **部署契约 (Deployment Contract)** | 👤 运维/DevOps | 提供容器/K8s/CI 真实配置 | 资源值、镜像、安全策略是否基于实际环境？ |

### 角色兜底原则

- **PM 兜底业务正确性**：AI 生成的代码是否符合业务预期？由 PM 在验收阶段最终确认。
- **TL 兜底架构安全性**：AI 生成的代码是否破坏现有系统？是否触碰红线？由 TL 在 Code Review 阶段最终确认。
- **后端开发兜底接口可用性**：AI 调用的接口是否存在？参数是否对齐？由后端开发在实现前确认。
- **测试工程师兜底质量**：AI 生成的测试是否真正覆盖业务分支？Mock 是否基于真实接口？由 QA 在测试 Review 阶段确认。
- **DevOps 兜底可部署性**：AI 生成的配置是否能直接部署？由 DevOps 在部署前确认。

## 2. Spec 编写流程（含人工审核节点）

### Step 0: AGENTS.md 自检（前置步骤 🆕）

> 💡 **V5.1 新增**：在开始任何 Spec 工作之前，AI 已自动读取项目根目录的 `AGENTS.md`。这包含项目的永久上下文——技术栈版本、代码规范、永久红线等。无需人工操作，但 TL 需确认 `AGENTS.md` 已完备。若不存在，先由 TL 按第 9 章模板创建。

### 关键规则

1. **未审核的 Spec 不可实现**：Spec 必须通过三道 Quality Gates 后，才能交给 AI 生成代码。
2. **未锚定的接口不可引用**：Spec 中出现的任何接口引用，必须有对应的 CodeAnchor 记录。
3. **未准备的上下文不可注入**：AI 实现前，Context Injection Checklist 必须全部打勾。

## 🧭 3.4 隐形边界声明 (Invisible Boundary Declaration)

> 👤 **填写人**: 技术负责人 (TL)
> 🔍 **审核人**: PM（确认业务容错策略）+ 后端开发（确认技术实现）

**Spec 定义了 "what"（要做什么），但 "how"（怎么做）的隐形约束决定了代码在生产环境是否能正确运行。本节声明所有"字面正确但隐含语义可能有误"的维度。**

### 3.4.1 幂等性 (Idempotency)

| 操作 | 幂等键 | 重复调用行为 | 说明 |
|:---|:---|:---|:---|
| [如：创建订单] | [如：out_trade_no] | [返回已有记录 / 幂等成功 / 报错] | [说明] |
| [如：发货操作] | [如：order_id + platform] | [跳过已处理] | [说明] |

### 3.4.2 并发安全 (Concurrency Safety)

| 共享资源 | 并发风险 | 锁策略 | 锁粒度 |
|:---|:---|:---|:---|
| [如：虚拟商品库存] | [超卖] | [数据库行级锁 (UPDATE WHERE stock >= ?)] | [单条记录] |
| [如：发货流水记录] | [重复插入] | [唯一索引] | [单条记录] |

### 3.4.3 分布式事务 (Distributed Transaction)

| 跨系统操作 | 事务边界 | 最终一致性方案 | 补偿机制 |
|:---|:---|:---|:---|
| [如：天猫发货 + 订单状态更新] | [本地事务表 + 外部 API] | [本地事务表 + 定时任务补偿] | [查询天猫发货状态，同步本地] |
| [如：发货成功 + MQ 通知] | [本地事务 + MQ 发送] | [RocketMQ 事务消息 / 本地消息表] | [未发送消息在事务回查时重发] |

### 3.4.4 错误分类 (Error Classification)

| 错误类型 | 示例错误码 | 处理策略 | 最大重试次数 | 重试间隔 |
|:---|:---|:---|:---|:---|
| 可重试错误 | `NETWORK_TIMEOUT`, `SYSTEM_BUSY` | 指数退避重试 | [N] | [间隔序列] |
| 参数错误 | `INVALID_PARAM`, `PRODUCT_NOT_EXIST` | 直接失败，人工介入 | 0 | — |
| 幂等成功 | `DUPLICATE_REQUEST` | 当作成功处理 | 0 | — |
| 限流错误 | `RATE_LIMITED` | 等待后重试 | [N] | [间隔序列] |

### 3.4.5 时间语义 (Time Semantics)

| 时间字段 | 存储类型 | 时区策略 | 比较方式 |
|:---|:---|:---|:---|
| [如：发货时间] | `Instant` (UTC 时间戳) | 数据库统一存储 UTC，展示层按用户时区转换 | 不做时区转换，直接比较时间戳 |
| [如：超时判断] | 相对时间（当前时间 - 事件时间 > 阈值） | 使用 `Instant.now()` 与数据库时间戳比较 | 差值比较，不依赖时区 |

### 3.4.6 数据安全 (Data Security)

| 敏感字段 | 脱敏策略 | 日志输出规则 | 存储加密 |
|:---|:---|:---|:---|
| [如：手机号] | 保留前 3 后 4，中间 **** 替换 | ✅ 脱敏后可输出 | — |
| [如：虚拟卡券密码] | 完全不可输出 | 🚫 严禁出现在日志中 | ✅ AES-256 加密 |
| [如：订单金额] | 保留完整值 | ✅ 可输出（非敏感） | — |

```
🔍 审核项：
☐ 每个涉及数据库写操作的步骤，幂等性是否已声明？
☐ 每个涉及共享资源修改的步骤，并发锁策略是否已声明？
☐ 每个跨系统操作（DB + API + MQ），分布式事务方案是否已声明？
☐ 每个调用外部接口的步骤，错误分类（可重试 vs 不可重试）是否已声明？
☐ 所有时间相关字段，时区策略是否已声明？
☐ 所有可能出现在日志中的数据，脱敏策略是否已声明？
```

## 🔗 3.5 技术契约 — 代码锚定 (CodeAnchor)

> 👤 **填写人**: 后端开发
> 🔍 **审核人**: TL（确认锚定准确性）

**这是 Spec 中最重要的字段。每一个逻辑接口必须锚定到代码库中真实存在的类和方法。**

| 编号 | 逻辑名称 | 真实类全路径 | 真实方法签名 | 返回值类型 | 所在模块 | 确认状态 | 确认人 |
|:---|:---|:---|:---|:---|:---|:---|:---|
| ANC-01 | [如：订单拆分] | `com.xxx.OrderSplitExecutor` | `executeSplit(String, List<Long>)` | `SplitResultVO` | `oms-order` | ✅ 已确认 | 张三 |
| ANC-02 | [如：平台发货] | `com.xxx.TmallAdapter` | `virtualShip(VirtualShipReq)` | `VirtualShipResp` | `platform-adapter` | ⬜ 待确认 | 李四 |

### 接口详情（每个 CodeAnchor 展开）

**ANC-01: [逻辑名称]**

```java
// 文件路径: [Git 仓库中的完整路径]
// 类定义:
public class [ClassName] {
    public [ReturnType] [methodName]([params]) throws [ExceptionType] {
        // 已有实现...
    }
}
```

```
🔍 审核项：
☐ 该接口是否在代码库中真实存在？（搜索验证）
☐ 方法签名是否与代码完全一致？（不可有任何偏差）
☐ 参数类型和返回值类型是否已确认？
☐ 该接口的调用是否需要特殊的前置条件？（如事务、鉴权）
```

### 每个 CodeAnchor 的 AI 注入清单

> ⚠️ **防幻觉规则**：AI 在实现阶段引用任何一个 CodeAnchor 时，必须将以下信息注入 AI 上下文窗口。缺少任何一项都会导致代码映射幻觉。

| 编号 | 必须注入的内容 | 注入形式 | 防幻觉作用 |
|:---|:---|:---|:---|
| ANC-XX | 完整方法签名（含 `throws` 声明） | 源文件片段 (Java/TS/Go) | 防止参数类型偏差和异常遗漏 |
| ANC-XX | 返回值类型完整定义 | 类定义源文件 (VO/DTO) | 防止返回值包装层错位 |
| ANC-XX | 所在类的 DI 注解 | `@Service` / `@Component` / `@Repository` | 防止依赖注入方式错误 |
| ANC-XX | 项目中已有的调用示例 | 至少 1 处真实调用代码片段 | 防止事务边界和异常处理错误 |

**注入示例（以 ANC-01 为例）：**

```java
// ===== 必须注入的上下文 START =====
// 1. 完整方法签名（含 throws）
@Service  // ← DI 注解
public class OrderSplitExecutor {
    public SplitResultVO executeSplit(String orderId, List<Long> skuIds) 
            throws OrderSplitException { ... }
}

// 2. 返回值类型完整定义
public class SplitResultVO {
    private boolean success;
    private String realOrderId;

## 5. TDD 执行规范

### 5.1 三回合原则

```
回合 1 (RED)：
  AI 只生成测试代码
  → 人运行 mvn test，确认编译通过且测试 RED
  → RED 原因必须为 "被测方法尚未实现"
  → 如果因编译错误 RED → 退回 AI 修正

回合 2 (GREEN)：
  AI 只生成实现代码
  → 人运行 mvn test，确认测试 GREEN
  → 如果测试失败 → 退回 AI 修正

回合 3 (REFACTOR)：
  AI 只做重构
  → 人运行 mvn test，确认测试仍 GREEN
  → 如果重构导致测试失败 → 退回 AI 修正
```

### 5.2 Mock 对象规则

- Mock 的类必须来自 CodeAnchor 中已确认的真实接口
- 不得 Mock 不存在的类（即 Spec 中臆造的接口名）
- 每个 Mock 的行为定义（when-thenReturn）必须符合该接口的真实语义

### 5.3 反模式警示

| 反模式 | 表现 | 正确做法 |
|:---|:---|:---|
| 🚫 "一次过" | AI 在一次响应中同时生成测试和实现 | 分两回合，人确认 RED 后再写实现 |
| 🚫 "自我验证" | AI 声称测试通过，但实际是假通过 | 人亲自运行 mvn test 并查看结果 |
| 🚫 "幻影 Mock" | Mock 了不存在的类 | 所有 Mock 必须来自 CodeAnchor |

## 7. 常见反模式与纠正

| 反模式 | 风险 | 纠正方法 |
|:---|:---|:---|
| **🚫 Spec 全由 AI 生成** | 接口臆造，与真实代码脱节 | AI 只辅助搜索接口，Spec 由人填写 |
| **🚫 跳过 CodeAnchor** | 接口名是臆造的，AI 生成的调用代码编译不过 | 每个接口引用必须有 CodeAnchor |
| **🚫 跳过 Quality Gates** | 未确认的 Spec 直接给 AI 实现 | 三道门禁全部签字后才能实现 |
| **🚫 AI 一次性生成全部代码** | 所有错误同时爆发，无法定位根因 | 按 Step 逐模块生成，逐模块验证 |
| **🚫 TDD 形同虚设** | 测试和实现由 AI 同次生成，"绿灯"是假的 | 严格分回合执行 RED → GREEN → REFACTOR |
| **🚫 配置使用占位符** | 生成的配置文件无法直接部署 | DevOps 提供真实配置值 |
